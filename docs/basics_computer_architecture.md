# 计算机基础：从晶体管到程序运行

本文从零开始，假设你只学过 C 语言基本语法（if/for/函数），除此之外没有任何计算机基础知识。用最通俗的语言解释程序到底是怎么在计算机里跑起来的。

---

## 1. 计算机是什么

### 1.1 一句话概括

**计算机 = CPU（计算） + 内存（记东西） + 输入输出设备（键盘/屏幕/硬盘/网卡）**

就像人一样：大脑负责想事情（CPU），记忆负责记东西（内存），嘴和耳朵负责交流（输入输出）。

### 1.2 计算机内部最简化的模型

```text
┌─────────────────────────────────────────────┐
│  主板（一块大电路板，把所有东西连起来）       │
│                                              │
│  ┌───────┐    ┌──────────────┐              │
│  │  CPU  │◄──►│  内存（RAM）  │              │
│  │       │    │ 8GB / 16GB   │              │
│  └──┬──┬─┘    └──────────────┘              │
│     │  │                                     │
│     │  └───────→ 显卡（GPU + 显存）           │
│     │                                        │
│     └─────────→ 硬盘（SSD）                   │
│                  网卡                        │
│                  USB / 键盘 / 鼠标            │
└─────────────────────────────────────────────┘
```

所有的东西都靠**总线**（Bus）通信——总线就是主板上的线路，想象成城市里的道路网。

### 1.3 CPU 内部长什么样

```text
CPU 内部：
┌────────────────────────────┐
│  控制单元（CU）              │  ← 负责"取指令、解码"，像指挥中心
│  ┌──────────────────────┐  │
│  │  寄存器（Registers）   │  │  ← CPU 内部的"超快小内存"
│  │  RAX, RBX, RCX, RDX   │  │     每个存 8 字节，总共几十个
│  │  RSP (栈指针)          │  │     比内存快几百倍
│  │  RIP (指令指针)         │  │
│  └──────────────────────┘  │
│  ┌──────────────────────┐  │
│  │  算术逻辑单元（ALU）   │  │  ← 负责加减乘除、比较大小
│  └──────────────────────┘  │
│  ┌──────────────────────┐  │
│  │  L1/L2/L3 高速缓存    │  │  ← CPU 内部的"缓存"，比内存快
│  └──────────────────────┘  │
└────────────────────────────┘
```

### 1.4 CPU 怎么执行程序——"取指-执行"循环

CPU 只会做一件极其简单的事，周而复始：

```text
无限循环 {
    1. 取指令：从内存中 RIP 指向的位置，取出下一条指令
    2. 解码：  看看这条指令是什么（加法？跳转？读内存？）
    3. 执行：  做指令要求的事情
    4. RIP++：  把指令指针移到下一条指令的位置
}
```

**每秒执行几十亿次这个循环。** 快到你根本感觉不到。

### 1.5 机器指令长什么样

这是 x86-64 指令（你的笔记本 CPU 用的）：

```asm
mov rax, [rbx]       ; 把内存中 rbx 地址处的内容读到 rax 寄存器
add rax, rcx          ; rax = rax + rcx
cmp rax, 100          ; 比较 rax 和 100
jg  somewhere         ; 如果大于就跳转到 somewhere
```

这些指令被编码成二进制数（比如 `mov rax, [rbx]` = `48 8B 03`），就是所谓的**机器码**。

你写的 C 代码最终都会被编译器翻译成这样的机器指令。

### 1.6 一个简单 C 程序的"CPU 视角"

```c
int add(int a, int b) {
    return a + b;
}
```

编译器把它翻译成（简化版）：

```asm
add:
    lea  eax, [rdi + rsi]   ; eax = rdi + rsi（a 在 rdi，b 在 rsi）
    ret                      ; 返回到调用者
```

当你调用 `add(3, 5)`：
1. CPU 把 3 放进 rdi 寄存器，5 放进 rsi 寄存器
2. CPU 跳转到 `add` 处的代码
3. CPU 执行 `lea eax, [rdi+rsi]` → eax = 8
4. CPU 执行 `ret` → 跳回调用处
5. `eax` 寄存器的值（8）就是返回值

## 2. 内存是怎么工作的

### 2.1 内存就是一大排编号的"储物格"

把内存想象成一长排信箱，每个信箱能存放 **1 个字节（8 个比特位）**：

```text
地址:    0    1    2    3    4    5    ...   8589934591
        ┌────┬────┬────┬────┬────┬────┬─────┬────┐
        │ 0A │ 3F │ 00 │ 00 │ 01 │ 00 │ ... │ 00 │
        └────┴────┴────┴────┴────┴────┴─────┴────┘
          ↑
        每个格子 1 字节（8 bit），能存 0~255（无符号）
```

- 你有 8GB 内存 → 约 85.9 亿个这样的格子
- 每个格子有唯一的**地址**（编号），从 0 到 85.9 亿
- **指针**就是一个格子的编号（地址）

### 2.2 为什么地址是 `0x7fff...` 这样的

`0x` 开头表示**十六进制**：
- 十进制：0, 1, 2, ..., 9, 10, 11, 12, 13, 14, 15
- 十六进制：0, 1, 2, ..., 9, A, B, C, D, E, F
- `0x7FFF` = 7×16³ + 15×16² + 15×16¹ + 15×16⁰ = 32767

地址只是一个大整数，计算机喜欢用十六进制显示而已。`0x7fff12345678` 翻译成十进制大约是 **140,000,000,000,000**，但这其实不重要——**你只需要知道"地址就是内存格子的编号"就够了。**

### 2.3 数据类型的本质：占几个格子

```text
char c;       // 1 个格子（1 字节），值范围 -128~127
int n;        // 4 个格子（4 字节），值范围 -21亿~21亿
long long x;  // 8 个格子（8 字节），值范围 -9×10¹⁸~9×10¹⁸
float f;      // 4 个格子，存的是浮点数（IEEE 754 格式）
double d;     // 8 个格子，双精度浮点数
void *p;      // 8 个格子（64 位系统），存的是地址
```

**数据类型 = 告诉编译器"从地址 N 开始，连续几个格子，以及这些格子里的数字怎么解释"。**

举例：内存地址 1000 开始存了 `00 00 80 3F`（4 个字节）
- 当 `int` 解读：`0x3F800000` = 1,065,353,216
- 当 `float` 解读：`1.0`
- 这就是为什么不能随便把一种类型的指针转成另一种

## 3. 操作系统是干什么的

### 3.1 为什么需要操作系统

**没有操作系统的世界**：你写的程序直接操作硬件。打印 `Hello World` 需要你自己写代码控制显卡输出像素，自己写代码读键盘，自己管理内存。

**有操作系统的世界**：操作系统（Windows/Linux）替你管理所有硬件，给你提供一个"幻觉"——你的程序独占整台电脑。

```text
你的程序                        现实
"我有 16GB 内存"               实际只给你分配了需要的部分
"我正在不间断运行"              操作系统在几百个进程间快速切换 CPU
"我读到了文件 /home/hello.txt"  操作系统帮你翻译成硬盘上的实际位置
```

### 3.2 三个关键抽象

**进程**：一个"正在运行的程序"。操作系统给每个进程分配独立的"虚拟地址空间"（让它以为有全部内存）。

**文件**：统一的操作接口。硬盘上的文件、键盘输入、网络数据、打印机……对操作系统来说都抽象成"文件"。

**虚拟内存**：每个进程看到的内存地址（如 `0x7fff12345678`）不是物理内存的真实地址。CPU 内部有个叫 MMU（内存管理单元）的硬件，自动把虚拟地址翻译成物理地址。

### 3.3 为什么这个项目里要区分"客户机"和"宿主机"

- **宿主机（Host）**：有真实 GPU 的机器，它的操作系统能通过驱动直接控制 GPU
- **客户机（Guest）**：没有 GPU 的机器（或故意不让它访问 GPU）

这个项目本质上是在客户机进程和宿主机进程之间，通过网络传输 CUDA 操作的请求和结果。

## 4. 程序是怎么从源码变成能运行的东西的

### 4.1 四个步骤

```text
hello.c（你写的文本）
   │
   │ ① 预处理（Preprocess）
   │   gcc -E hello.c -o hello.i
   │   把 #include 的文件内容复制进来，#define 替换
   v
hello.i（展开后的文本，可能有几万行）
   │
   │ ② 编译（Compile）
   │   gcc -S hello.i -o hello.s
   │   把 C 代码翻译成汇编代码
   v
hello.s（汇编代码，人类勉强能读）
   │
   │ ③ 汇编（Assemble）
   │   gcc -c hello.s -o hello.o
   │   把汇编代码翻译成机器码（二进制）
   v
hello.o（目标文件，二进制）
   │
   │ ④ 链接（Link）
   │   gcc hello.o -o hello
   │   把 .o 文件和它依赖的库合并成可执行文件
   v
hello（可执行文件，可以直接运行）
```

### 4.2 为什么需要"链接"这一步

你的程序调用 `printf`，但 `printf` 的代码不在你的 `hello.c` 里。`printf` 的实现在 C 标准库（`libc.so`）里。

**链接器的工作**：找到 `printf` 的实际代码在哪里，把调用 `printf` 的地方填上正确的地址。就像一个编辑把所有"参见第 X 页"的引用填上实际页码。

### 4.3 动态链接（跟这个项目直接相关）

回顾基础文档里的内容——**动态链接**就是把"填页码"这件事推迟到程序运行时再做：

```text
编译时：  hello 里只记录"我需要 printf，在 libc.so 里"
运行时：  动态链接器加载 libc.so，找到 printf，填上真实地址
```

`LD_PRELOAD` 就是在这个"运行时找函数"的阶段动手脚——在找真实 `libcudart.so` 之前，先加载你的 fake 版本。

## 5. 从晶体管到 `cudaMalloc`：完整调用链

以 vGPU 项目的核心操作为例，看看到底发生了什么：

```text
你在 client 代码中写：
    cudaMalloc(&d_ptr, 1048576)
         │
         │ ① C 编译器翻译成
         v
    call cudaMalloc@PLT    （x86 汇编指令）
         │
         │ ② CPU 执行这条 call 指令
         v
    跳转到 libcudart_proxy.so 中的 cudaMalloc 函数
         │
         │ ③ 你的代码执行
         v
    创建 MallocRequest protobuf 对象
    req.set_size(1048576)
         │
         │ ④ protobuf 库把 1048576 编码成二进制
         v
    08 00 10 00 00 10 ... （二进制字节流）
         │
         │ ⑤ gRPC 库调用 TCP socket write
         v
    操作系统把字节流复制到内核网络缓冲区
         │
         │ ⑥ 网卡驱动把数据通过"网络"发送
         v
    127.0.0.1 回环 → 直接到达本机网络接收缓冲区
         │
         │ ⑦ server 进程的 gRPC 收到数据
         v
    gRPC 反序列化 → MallocRequest 对象
         │
         │ ⑧ server 调用真实 CUDA
         v
    cuMemAlloc(&real_ptr, 1048576)
         │
         │ ⑨ CUDA 驱动通过 ioctl 和 GPU 通信
         v
    GPU 在显存中分配 1MB 空间，返回真实 GPU 地址
         │
         │ ⑩ server 生成虚拟指针，打包成 MallocReply
         v
    网络回传 → gRPC 反序列化 → client 拿到 vptr
```

**你写的 `cudaMalloc` 一行代码，触发了这么多层的调用。** 但每一层只做一件简单的事，层层叠加起来就完成了复杂的功能。
